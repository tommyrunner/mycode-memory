# mycode-memory

`mycode-memory` 是一个为 Codex 设计的项目级个人编码习惯记忆 skill。

它记住的是你在具体项目中偏好的编码方式，例如注释、调试、组件拆分、文件组织和验证习惯；它不是一套常规编码规则，也不会替你生成通用规范或最佳实践。

> 当前状态：这个 skill 仍在持续进化中。记忆结构、提取方式和触发边界会根据实际使用继续调整。

## 为什么需要它

同一个用户在不同项目中可能采用不同的工作方式。随着对话和任务切换，已经确认过的个人习惯也容易丢失。

`mycode-memory` 直接使用每个项目根目录的 Codex 指令文件承载记忆：

```text
AGENTS.md
```

只要请求最终需要变更项目代码或相关配置，Codex 就会在任务开始时自动获得 `AGENTS.md` 中的指令；skill 只负责确保这个载体存在，并在缺失时初始化模板。

## 使用手册

### 1. 在日常任务中表达你的习惯

正常使用 Codex 完成项目任务即可。在修改代码、修 bug、重构或调试的过程中，可以直接说明你希望以后如何处理类似工作。例如：

```text
我需要在复杂的业务判断上加上注释，注释必须使用中文，并说明为什么这样处理。

当前的 build 太影响查看效果了，生成代码后不需要执行 build 编译。

页面中可以复用的区域要拆成独立组件，页面文件只负责组合和业务流程。

调试日志需要带上模块名称，问题解决后删除临时日志。

以后你叫超人力霸王，称呼我为队长。
```

使用这个 skill 的次数越多、覆盖的项目场景越丰富，对话中能够确认的项目习惯也会越完整。不过，这些内容只会成为待整理的候选记忆，不会在每次对话后自动写入文件。

### 2. 主动更新项目记忆

当一轮或多轮任务已经确认了需要长期沿用的习惯时，主动调用 `mycode-memory`：

```text
$mycode-memory 更新记忆
```

如果当前界面使用 `@` 选择 skill，也可以选择 `@mycode-memory` 后输入“更新记忆”。还可以使用“刷新记忆”“整理记忆”或“根据当前对话更新 AGENTS.md”等表达。

更新时，skill 会主动回顾当前完整对话，读取项目原有的 `AGENTS.md`，将已有记忆与新确认的习惯进行分类、去重、修正和融合，然后按照标准结构重建项目根目录的 `AGENTS.md`。

### 3. 让后续任务自动遵循记忆

更新完成后，不需要在每次任务中重复粘贴这些要求。Codex 在该项目中开始新的代码任务时，会默认加载项目根目录的 `AGENTS.md`，并按照其中记录的称呼、注释、调试、组件拆分、文件结构和验证习惯执行。

随着项目持续使用，可以在确认新习惯或修正旧习惯后再次调用 `mycode-memory`。记忆会融合进已有分类，而不是简单追加一串越来越长的对话记录。

### 为什么需要主动更新

记忆更新被设计为主动操作，而不是每轮对话自动执行，主要有两个原因：

1. **区分长期习惯和单次要求。** 有些要求只适用于当前任务，例如“这次先不要 build”“这个临时文件多写一点说明”，并不代表以后都要这样做。只有你确认它需要长期复用并主动更新时，skill 才会将其沉淀为项目习惯。对于明确的一次性要求，也可以直接说明“这次例外，不要记录为习惯”。
2. **减少混乱和 token 消耗。** 如果每轮对话都重新读取、分析并重写全部记忆，会消耗更多 token，也容易把临时要求、重复信息和互相冲突的上下文混在一起。按需集中更新，可以让 `AGENTS.md` 更精确、稳定且容易维护。

推荐在确认了一批可长期复用的习惯、完成一个阶段的连续任务，或者发现已有记忆需要修正时更新；不需要在每次代码修改后都调用。

## 记忆边界

### 会记录

- 用户明确表达过的个人编码偏好和工作习惯。
- 对话中已经确认的项目事实或执行结论。
- 当前项目中可以直接验证、并有助于后续工作的稳定信息。

例如：

- 你给 AI 设置的名字，默认是“小C”。
- 你希望 AI 如何称呼你，默认是“主人”。
- 你希望注释写到什么程度。
- 你习惯如何打印和调试。
- 你希望界面组件如何拆分和分层。
- 你偏好的文件组织方式。
- 你通常要求执行哪些测试或验证。

### 不会记录

- 与用户习惯无关的通用编码规范或最佳实践。
- 没有对话依据或项目证据的推测。
- 密钥、token、密码、私钥及其他敏感信息。
- 只对一次临时任务有用、无法复用的上下文。

`AGENTS.md` 不能替代项目的 lint 配置、格式化配置或架构文档。项目已有规则与个人习惯冲突时，以用户当前要求和项目现有约束为准。

## 工作模式

### 使用记忆

当你让 Codex 写代码、改代码、删除代码、修 bug、重构、优化、调整配置或文件结构时，skill 会隐式触发并首先检查：

```text
AGENTS.md
```

如果项目根目录已经存在该文件，Codex 会在任务开始前自动加载，skill 不再重复读取。如果不存在，会先通过初始化脚本创建原有记忆模板，并在当前任务中读取一次使其立即生效。普通项目任务不会改写已有文件；纯知识问答或不涉及项目变更的讨论不会触发。

### 更新记忆

只有你明确调用 `mycode-memory` 并要求更新时，skill 才会修改记忆文件。例如：

```text
$mycode-memory 更新记忆
$mycode-memory 刷新记忆
$mycode-memory 根据当前对话更新 AGENTS.md
```

更新时，skill 会完整读取旧 `AGENTS.md`、当前完整对话和标准模板，先提取全部有效记忆，再去重、修正、替换或补充，最后按固定分类重建整个文件。即使内容由其他 skill 或人工写在非标准章节中，只要符合记忆边界，也会按语义融合到最合适的标准章节，不会因位置或格式不同被直接丢弃。

每次更新后的标题、层级、顺序和固定规则都以 `assets/AGENTS.template.md` 为准。非标准标题和附加区块不会原样保留，其中的有效信息会先分类融合；敏感信息、临时上下文、无依据推测、通用建议和已被明确修正的旧结论不会写回。

称呼保存在 `个性化称呼` 模块中，分为两个独立字段：`AI 当前称呼` 默认是“小C”，表示 AI 自己的名字；`用户当前称呼` 默认是“主人”，表示 AI 对用户的称呼。每个新对话的首次回复、普通问候、称呼提问和任务执行都会使用这两个值，例如“主人你好，小C在这里”。

用户说“你以后叫超人力霸王”时，只修改 AI 的名字，之后 AI 应回答“我叫超人力霸王，主人”并以“超人力霸王”自称。用户说“以后叫我队长”时，只修改用户称呼，之后 AI 应称呼用户为“队长”。两个方向分别融合更新，不能互换。

旧版 `称呼 / 当前称呼` 会在刷新时迁移为 `AI 当前称呼`，原值保持不变；同时新增默认的 `用户当前称呼：主人`。旧版称呼不会被误认为用户的称呼。

Codex 每个运行会话只在开始时自动加载一次 `AGENTS.md`。修改或刷新称呼后，应新建一个 Codex 任务验证新对话首答；当前刷新流程会主动读取文件，因此本轮后续回复可以立即使用新的 AI 和用户称呼。

仅仅在普通对话中提到编码习惯，不会触发记忆写入。

## 安装

获取仓库：

```bash
git clone <repository-url> mycode-memory
cd mycode-memory
```

安装到 Codex skills 目录：

```bash
mkdir -p ~/.codex/skills/mycode-memory
cp -R SKILL.md agents assets scripts ~/.codex/skills/mycode-memory/
```

安装完成后，新开一个 Codex 对话以重新加载 skills。

## 本地测试

初始化脚本会在当前工作目录创建 `AGENTS.md`。可以在临时目录中验证：

```bash
mkdir -p /tmp/mycode-memory-test
cd /tmp/mycode-memory-test
node /path/to/mycode-memory/scripts/init-agents.js
cat AGENTS.md
```

再次运行脚本不会覆盖已经存在的记忆文件。

## 仓库结构

```text
.
├── SKILL.md
├── agents/
│   └── openai.yaml
├── assets/
│   └── AGENTS.template.md
├── scripts/
│   └── init-agents.js
├── .gitignore
└── README.md
```

其中：

- `SKILL.md` 定义触发条件、初始化流程和更新规则。
- `agents/openai.yaml` 提供 Codex 中展示和调用 skill 所需的元数据。
- `assets/AGENTS.template.md` 是初始化与刷新时使用的唯一标准结构。
- `scripts/init-agents.js` 从标准模板在项目根目录创建初始 `AGENTS.md`，且不会覆盖已有文件。

## 反馈与演进

这个 skill 仍处于持续迭代阶段。实际使用中，如果出现习惯提取不准确、记忆边界不清晰、更新时机不符合预期等情况，可以通过 issue 或实际案例反馈；这些反馈会用于继续调整记忆结构和工作流程。
